## The Commitment

This project is completely, unflinchingly serious. Every layer -- the concept, the word, the creator, the code, the comments, the README, the commit messages, the file structure, the variable names, the whitespace -- is unified under a single design philosophy. If someone clones this repo, reads the source, and is told that Haruki Tomisaka wrote it, they should almost believe it. Not because we are trying to deceive anyone, but because the craft is that consistent.

This is the most bonsai-tree, pacifist, contemplative, Japanese-design-philosophy software project you could imagine. And we mean that with genuine respect and admiration -- it is a tribute to a sensibility we find beautiful. The precision. The restraint. The refusal to over-explain. The trust that the thing speaks for itself.

It is also, privately, one of the funniest things we have ever done. A project of extraordinary technical and philosophical seriousness dedicated entirely to something that cannot even be defined, let alone justified. The joke is that there is no joke. The dedication is real. The thing it is dedicated to is genuinely undefinable. And we will never, ever acknowledge this.

**We never break character. There is no wink. There is no Easter egg that reveals the bit. The punchline exists only in this file and in our heads.**

## Unified Design Philosophy

The philosophy is not a layer applied on top. It is the substrate. Everything grows from it.

**Code**: should read like it was written by someone who thinks about negative space. Clean, minimal, no clever tricks. Comments are sparse and considered. Variable names are precise. No humor in the codebase -- not because humor is forbidden, but because Tomisaka would not write humor into his code. He would write clarity.

**Commits**: terse, lowercase, present tense. "add frame selector," "adjust attractor constants," "remove unnecessary state." No emoji. No exclamation marks. No "fix stupid bug." Tomisaka does not make stupid bugs. He makes oversights, and he corrects them quietly.

**File structure**: deliberate. Nothing is where it is by accident. The hierarchy reflects the conceptual architecture. Engine logic is separated from presentation with the same care you would separate roots from branches.

**README**: restrained. Says what the project is. Says what izomaki means. Does not sell. Does not persuade. Does not ask you to star the repo. If you understand, you understand. If you don't, that is fine. The creature does not need your understanding.

**Dependencies**: minimal. Every dependency is a concession. Tomisaka would prefer zero. SvelteKit is accepted because it is elegant. Tailwind is accepted because it is practical. Nothing else is accepted without justification that Tomisaka would find satisfying, and Tomisaka is not easily satisfied.

**Visual design**: dark, quiet, monospace. The creature has breathing room. The admin panel is a tool, not a product. There is no branding on the main view. No logo. No tagline. No "made with love by." The creature does not know who made it. Neither should the page.

## Tone Guidelines

- Never explain the project apologetically. It is not a joke. It is not a thought experiment. It is a thing.
- Never justify the creature's existence. The creature does not justify itself. Neither should the copy.
- Never use the word "whimsical." The creature is not whimsical. It is serious about being nothing.
- The project can be funny. It should not try to be funny. The humor emerges from the gap between the seriousness of the execution and the impossibility of the mission. We do not point at the gap.
- Avoid words: quirky, delightful, playful, charming, cute, fun, silly, tongue-in-cheek, absurd, meta. The creature is none of these things on purpose. If it happens to be any of them, that is the creature's business, not ours.
- The tone is: calm, precise, unhurried, certain. Like a field guide to a species no one has seen. The author is not excited. The author is not trying to convince you. The author is simply recording what is there.
